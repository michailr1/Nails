# Архитектура

## 1. Цель первого этапа

Предоставить мастеру закрытого Telegram-помощника для ежедневного ведения записей без необходимости вручную работать с календарём и отдельной CRM.

## 2. Компоненты

### Закрытый Telegram-бот

- принимает сообщения только от разрешённых Telegram ID;
- связывает подтверждённый Telegram ID с ролью `master` или `admin`;
- передаёт идентификатор пользователя как доверенный transport context, а не как аргумент модели;
- поддерживает кнопки подтверждения для опасных или неоднозначных операций.

### Hermes

- отдельный профиль проекта;
- отдельный skill `nails-scheduling`;
- распознаёт намерение и параметры запроса;
- задаёт уточняющие вопросы;
- вызывает только разрешённые функции Booking API;
- не выполняет произвольные SQL-запросы и не изменяет Google Calendar напрямую.

### Booking API

Единственный компонент, который применяет бизнес-правила:

- проверка ролей;
- поиск свободных интервалов;
- проверка пересечений;
- расчёт длительности и буферов;
- создание, перенос и отмена записей;
- управление карточками клиентов;
- фиксация цены на момент записи или завершения визита;
- аудит изменений;
- идемпотентность повторных запросов;
- постановка задач одностороннего экспорта в Google Calendar.

### PostgreSQL

Единственный источник истины для расписания и бизнес-данных. Хранит:

- пользователей и роли;
- каталог услуг;
- клиентов и внутренние алиасы;
- записи и снимки стоимости;
- рабочий график и исключения;
- аудит;
- задания синхронизации с Google Calendar;
- связи с созданными событиями календаря.

### Google Calendar

Используется только как дополнительное визуальное представление подтверждённого расписания.

- синхронизация односторонняя: из PostgreSQL в календарь;
- бизнес-операция считается успешной после фиксации в PostgreSQL;
- после изменения записи Booking API создаёт задание в `calendar_sync_jobs`;
- фоновый worker выполняет экспорт с повторами и увеличивающейся задержкой;
- после исчерпания повторов администратор получает уведомление;
- фоновая сверка обнаруживает и исправляет рассинхронизацию;
- недоступность календаря не блокирует создание, перенос и отмену записей;
- секреты интеграции доступны только компоненту синхронизации.

Интеграция подключается на позднем этапе MVP (NAILS-006), после проверки основного контура записи в живой работе.

## 3. Поток создания или изменения записи

```text
Сообщение мастера
  -> определение операции и параметров
  -> проверка клиента, услуги и времени
  -> подтверждение итоговых параметров
  -> транзакция Booking API в PostgreSQL
       - изменение записи
       - аудит
       - calendar_sync_job (после NAILS-006)
  -> успешный ответ мастеру
  -> фоновый экспорт в Google Calendar
```

Google Calendar не участвует в транзакции создания, переноса или отмены записи.

## 4. Время и часовой пояс

- всё расписание ведётся в одном локальном часовом поясе мастера;
- зона задаётся обязательным параметром `APP_TIMEZONE` в формате IANA, например `Europe/Paris`;
- пустая, неизвестная или некорректная зона блокирует запуск приложения;
- автоматический переход к UTC запрещён;
- пользовательский ввод и ответы бота интерпретируются в выбранной зоне;
- в БД моменты времени могут храниться как timezone-aware timestamps, но бизнес-границы дня вычисляются в зоне мастера;
- переходы на летнее и зимнее время покрываются тестами.

## 5. Резервное копирование

Система заменяет записную книжку, поэтому сохранность данных — требование первого рабочего среза:

- регулярные автоматические бэкапы PostgreSQL;
- хранение копий вне рабочей БД и вне GitHub;
- журнал успешности и ошибок резервного копирования;
- периодическая проверка фактического восстановления;
- пилот не считается готовым к основному использованию без успешного тестового восстановления.

## 6. Обратная связь и защищённые логи

- под ответами бота может быть кнопка «не то» (👎);
- сохраняются только последние несколько сообщений, необходимые для разбора;
- перед сохранением удаляются секреты и по возможности маскируются контакты;
- срок хранения настраивается, начальное значение — 30 дней;
- доступ к содержимому имеет только роль `admin`;
- событие можно удалить вручную;
- журнал нераспознанных запросов подчиняется тем же правилам;
- реальные диалоги никогда не попадают в GitHub или CI-логи.

## 7. Фактическое отклонение от расписания

Если процедура задерживается или начинается позже:

- мастер может указать ожидаемое новое время окончания;
- система показывает возможный конфликт со следующей записью;
- последующие записи не сдвигаются автоматически;
- решение о переносе принимает мастер;
- фактическое начало и окончание могут сохраняться отдельно от плановых значений.

## 8. Минимальные метрики пилота

Для решения о переходе со старой записной книжки фиксируются агрегированные показатели:

- успешно созданные записи;
- исправленные или отменённые операции;
- повторные уточнения агента;
- нажатия 👎 и нераспознанные запросы;
- расхождения со старым расписанием;
- ошибки резервного копирования;
- ошибки синхронизации календаря после его подключения.

Метрики не должны содержать лишние персональные данные.

## 9. Принципы безопасности

- профиль Hermes не считается самостоятельной песочницей;
- модель не получает credentials Google Calendar;
- backend не доверяет идентификаторам и ролям из текста сообщения;
- все write-операции проверяются сервером;
- отмена, перенос и удаление требуют подтверждения;
- повторное нажатие не должно создавать дубликат;
- production-данные не попадают в GitHub.

## 10. Будущий публичный контур

Публичный бот для клиентов будет отдельным изолированным компонентом и получит существенно урезанные полномочия. Он не входит в первый этап и не должен запускаться в том же доверенном контуре, что и агент мастера.
