Date: **2026-07-14**.

Status: **deployment candidate; production not yet changed**.

## Production defect

Manual Telegram acceptance of NAILS-002E4 found two blocking problems:

1. the model converted “пятница” into the wrong absolute date before calling backend scheduling operations;
2. a completed onboarding interview had to be restarted to correct one mistaken working date because scheduling `0.1.0` had no availability write action.

Correct calendar interpretation:

```text

2026-07-15 — Wednesday

```

Correct desired data after a confirmed Telegram correction:

```text

2026-07-15 11:00–20:00 — unchanged
2026-07-17 11:00–15:00 — added
2026-07-18 — removed as unknown, not marked as a day off
```


The runbook creates:

- a compressed PostgreSQL dump under `/opt/nails/backups`;
- a root-only Hermes runtime/config/skill/image metadata backup under the Nails profile.

It then:

1. fast-forwards to an exact approved release SHA containing the reviewed fix;
2. builds a new API image while the old API remains available;
3. stops only the root user-level Nails gateway;
4. recreates only `nails-api` with `--no-deps`;
5. proves `nails-db` and Docker daemon identity/start time are unchanged;
6. verifies Alembic remains `0006` and both new OpenAPI routes exist;
7. installs scheduling plugin `0.2.0` and updated onboarding/scheduling skills;
8. proves Hermes config is byte-for-byte unchanged;
9. verifies plugin list, registry, Telegram visibility and new actions;
10. restarts only the root user-level gateway and scans logs.

Rollback restores:

- previous repository HEAD;
- previous scheduling plugin and both skills;
- previous Hermes config;
- previous API image reference and API container;
- active gateway state.
