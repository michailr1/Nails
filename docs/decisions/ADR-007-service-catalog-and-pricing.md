# ADR-007: каталог услуг с допами, типы цен и процесс закрытия визита

- Статус: принято
- Дата: 2026-07-19

## Контекст

Реальный прайс мастера (фото прайс-листов «Маникюр» и «Педикюр») не сводится к модели «одна услуга = одна железная цена и длительность»:

- **базовые услуги** — почти фиксированная цена (Классический 1200);
- **диапазоны** — «Педикюр 1900–2100, зависит от состояния стоп»;
- **допы, которые складываются** — снятие +100, зачистка +300, парафин +500;
- **цена за единицу** — «дизайн 50/100 за 1 ноготь»;
- итоговые **цена и длительность визита зависят от набора допов и конкретной клиентки** и окончательно известны только после работы.

Прайс планируется показывать клиенткам (клиентский контур ADR-004, web-экспорт), поэтому каталог обязан выражать полную структуру прайса уже сейчас — допы нельзя держать только заметкой в записи.

Две крайности отвергнуты: «занести весь прайс железными ценами» не выражает диапазоны и комбинации и утомляет мастера; «убрать цены и спрашивать каждый раз» теряет каталог и заставляет вводить число на каждой записи.

## Решение

### 1. Каталог: одна таблица, два вида позиций

- `kind = base` — записываемая услуга; несёт длительность-ориентир.
- `kind = addon` — доп; отдельно не записывается, прикрепляется к записи; несёт добавку времени `extra_minutes` (может быть 0).
- Тип цены (`price_type`) — ровно четыре формы из реального прайса:
  - `fixed` — 1200;
  - `range` — min–max (1900–2100), с примечанием («зависит от состояния»);
  - `per_unit` — сумма за единицу («за 1 ноготь»);
  - `on_request` — цены нет, спрашиваем.
- `category` (Маникюр / Педикюр / Дополнительно / Дизайн / Парафинотерапия) и порядок сортировки — чтобы отрендерить публичный прайс в структуре оригинала.

Система модификаторов и универсальные правила сочетаний допов не строятся. Дополнение прибавляет цену и время, кроме узких явно сохранённых связей «время дополнения уже включено в стандартный слот этой основной процедуры».

### 2. Цены и длительности — ориентиры, не гейты

По принципу §3.11 (открыто по умолчанию): каталожные цена и длительность **предзаполняют** запись, мастер **одним словом переопределяет** любую из них («по деньгам 2200», «ставь 2.5 часа»). Отсутствие цены (`on_request`) — не блокер записи, а причина спросить.

### 3. Процесс записи с допами

1. Мастер: «запиши Лену завтра на 15, маникюр с покрытием, и ноготь отремонтировать».
2. Ассистент: карточка клиентки (exact/candidates) → база + допы из каталога → **время** = длительность базы + добавки допов, не включённые в стандартный слот базы (проверка дня по ADR-006: выходной/overlap) → **цена** = сумма с учётом количества для `per_unit` (для range — вилка).
3. Одна сводка «сейчас → будет» с составом, количеством, временем и ориентиром цены → явное подтверждение → guarded write.
4. Снапшот в записи: состав (база + допы), количество, признак включённого времени и цены на момент записи; для range пишется вилка, не выдуманное число.

### 4. Закрытие визита: вечерний дайджест

- **Один раз в сутки, вечером** (время настраивается мастером, дефолт ~21:30): ассистент присылает один вопрос по визитам дня, чьё время окончания прошло и которые не закрыты: «Всё прошло хорошо? Какие итоговые цены? Можно не отвечать — останутся ориентиры».
- **Молчание — валидный ответ**: остаётся зафиксированный ориентир; для range фиксируется нижняя граница с пометкой «не уточнено». Повторно на следующий день не переспрашивается.
- Финализация мягкая: мастер в любой момент может поправить словами («у Лены вчера вышло 2400»).
- «Не пришла» в ответе → no-show (NAILS-005).
- Ещё не завершённый поздний визит в сегодняшний дайджест не попадает и уходит в следующий вечер.
- Механика: systemd-таймер на VPS (прецедент — backup-таймер) + отправка по образцу `feedback_tool` с токеном **только из Hermes profile `.env`**. Backend токен не получает. Одно сообщение в сутки.

Выручка и отчёты считаются по фактическим ценам записей.

### 5. Занесение прайса по фото

- В профиле уже разрешён vision: мастер присылает фото прайса, ассистент распознаёт позиции, цены и категории.
- Длительностей в прайсе нет — ассистент **сам предлагает** прикидки в отдельной колонке «моя прикидка», помеченной как предложение; мастер правит только несогласное.
- Занесение — **одна подтверждённая batch-мутация** (одна таблица, один цикл правок, одно «да»), не интервью по каждой позиции.

### 6. Отложено осознанно

- персональные цена/длительность под клиентку — NAILS-004 (client overrides), сейчас закрывается override'ом мастера при записи;
- сверка каталожных длительностей с фактическими и предложения обновления — после NAILS-005 (метрики);
- публичный рендер прайса клиенткам — клиентский контур ADR-004 / web-экспорт, отдельные слайсы;
- универсальная матрица применимости и модель вариантов — до подтверждённого массового сценария.

### 7. Калибровка по ответам пилотного мастера

Стандартные интервалы процедуры и время после неё:

| Основная процедура | Процедура | После процедуры |
| --- | ---: | ---: |
| Маникюр | 60 мин | 5 мин |
| Маникюр с покрытием | 130 мин | 5 мин |
| Педикюр | 90 мин | 10 мин |
| Педикюр с покрытием | 120 мин | 10 мин |

Подтверждены ровно три связи включённого времени:

- «Маникюр с покрытием» ↔ снятие;
- «Маникюр с покрытием» ↔ укрепление;
- «Педикюр с покрытием» ↔ снятие.

Укрепление для педикюра не предполагается. Та же позиция дополнения на другой основной процедуре продолжает прибавлять своё время.

Дополнительное время сверх стандартного слота:

- френч — 20 минут;
- простой дизайн — 10 минут;
- ремонт — 10 минут за ноготь;
- наращивание — 20 минут за ноготь.

Количество для `per_unit` сохраняется в записи, по умолчанию равно 1 и масштабирует время. Для ремонта оно масштабирует также цену; цена наращивания может оставаться `on_request`. Цена любого педикюра зависит от состояния стоп, но это само по себе не создаёт вымышленную верхнюю границу: используются только фактически заданные `range`, «от» или `on_request` значения. Цифры цен этой калибровкой не изменяются.

## Инварианты (не ослабляются)

Подтверждение мастером каждой мутации (batch-занесение — одна подтверждённая мутация); снапшоты цен, количества и состава в записях; owner-scoping; честность ассистента (`ok=true`/`verified=true`); токен Telegram — только в profile `.env`.
