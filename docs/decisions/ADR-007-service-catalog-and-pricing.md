# ADR-007: каталог услуг с допами, типы цен и процесс закрытия визита

- Статус: принято
- Дата: 2026-07-19

## Контекст

Реальный прайс мастера (фото прайс-листов «Маникюр» и «Педикюр») не сводится к модели «одна услуга = одна железная цена и длительность»:

- **базовые услуги** — почти фиксированная цена (Классический 1200);
- **диапазоны** — «Педикюр 1900–2100, зависит от состояния стоп»;
- **допы, которые складываются** — снятие +100, зачистка +300, парафин +500;
- **цена за единицу** — «дизайн 50/100 за 1 ноготь»;
- итоговые **цена и длительность визита зависят от набора допов и конкретной клиентки** и окончательно известны только после работы.

Прайс планируется показывать клиенткам (клиентский контур ADR-004, web-экспорт), поэтому каталог обязан выражать полную структуру прайса уже сейчас — допы нельзя держать только заметкой в записи.

Две крайности отвергнуты: «занести весь прайс железными ценами» не выражает диапазоны и комбинации и утомляет мастера; «убрать цены и спрашивать каждый раз» теряет каталог и заставляет вводить число на каждой записи.

## Решение

### 1. Каталог: одна таблица, два вида позиций

- `kind = base` — записываемая услуга; несёт длительность-ориентир.
- `kind = addon` — доп; отдельно не записывается, прикрепляется к записи; несёт добавку времени `extra_minutes` (может быть 0).
- Тип цены (`price_type`) — ровно четыре формы из реального прайса:
  - `fixed` — 1200;
  - `range` — min–max (1900–2100), с примечанием («зависит от состояния»);
  - `per_unit` — сумма за единицу («за 1 ноготь»);
  - `on_request` — цены нет, спрашиваем.
- `category` (Маникюр / Педикюр / Дополнительно / Дизайн / Парафинотерапия) и порядок сортировки — чтобы отрендерить публичный прайс в структуре оригинала.

Система модификаторов, правила сочетаний допов и моделирование «времени за ноготь» **не строятся** (несоразмерно одному мастеру): доп просто прибавляет цену и время.

### 2. Цены и длительности — ориентиры, не гейты

По принципу §3.11 (открыто по умолчанию): каталожные цена и длительность **предзаполняют** запись, мастер **одним словом переопределяет** любую из них («по деньгам 2200», «ставь 2.5 часа»). Отсутствие цены (`on_request`) — не блокер записи, а причина спросить.

### 3. Процесс записи с допами

1. Мастер: «запиши Лену завтра на 15, маникюр с покрытием, и ноготь отремонтировать».
2. Ассистент: карточка клиентки (exact/candidates) → база + допы из каталога → **время** = длительность базы + добавки допов (проверка дня по ADR-006: выходной/overlap) → **цена** = сумма (для range — вилка).
3. Одна сводка «сейчас → будет» с составом, временем и ориентиром цены → явное подтверждение → guarded write.
4. Снапшот в записи: состав (база + допы) и цены на момент записи; для range пишется вилка, не выдуманное число.

### 4. Закрытие визита: вечерний дайджест

- **Один раз в сутки, вечером** (время настраивается мастером, дефолт ~21:30): ассистент присылает один вопрос по визитам дня, чьё время окончания прошло и которые не закрыты: «Всё прошло хорошо? Какие итоговые цены? Можно не отвечать — останутся ориентиры».
- **Молчание — валидный ответ**: остаётся зафиксированный ориентир; для range фиксируется нижняя граница с пометкой «не уточнено». Повторно на следующий день не переспрашивается.
- Финализация мягкая: мастер в любой момент может поправить словами («у Лены вчера вышло 2400»).
- «Не пришла» в ответе → no-show (NAILS-005).
- Ещё не завершённый поздний визит в сегодняшний дайджест не попадает и уходит в следующий вечер.
- Механика: systemd-таймер на VPS (прецедент — backup-таймер) + отправка по образцу `feedback_tool` с токеном **только из Hermes profile `.env`**. Backend токен не получает. Одно сообщение в сутки.

Выручка и отчёты считаются по фактическим ценам записей.

### 5. Занесение прайса по фото

- В профиле уже разрешён vision: мастер присылает фото прайса, ассистент распознаёт позиции, цены и категории.
- Длительностей в прайсе нет — ассистент **сам предлагает** прикидки в отдельной колонке «моя прикидка», помеченной как предложение; мастер правит только несогласное.
- Занесение — **одна подтверждённая batch-мутация** (одна таблица, один цикл правок, одно «да»), не интервью по каждой позиции.

### 6. Отложено осознанно

- персональные цена/длительность под клиентку — NAILS-004 (client overrides), сейчас закрывается override'ом мастера при записи;
- сверка каталожных длительностей с фактическими и предложения обновления — после NAILS-005 (метрики);
- публичный рендер прайса клиенткам — клиентский контур ADR-004 / web-экспорт, отдельные слайсы.

### 7. Калибровка по ответам мастера (пилот, 2026-07-22)

Первый разбор реального прайса с мастером подтвердил направление ADR (ориентиры, не гейты; открыто по умолчанию; финализация вечером) и дал конкретные калибровки. Это не пересмотр решения, а его настройка на фактические данные.

- **Базовые ориентиры длительности** — мастер ставит стандартное время на клиентку и не допрашивает каждый раз: обычный маникюр 60, маникюр с покрытием 130, обычный педикюр 90, педикюр с покрытием 120 минут. Это дефолты `duration_minutes`, переопределяемые словом.
- **Буфер уборки — категорийный дефолт, не ноль**: после маникюра 5 минут, после педикюра 10. `buffer_after_minutes` предзаполняется по разделу (уточняет product-principles §5), мастер по-прежнему может обнулить или изменить.
- **Педикюр — цена по осмотру, не `fixed`**: любая педикюрная база уточняется по состоянию стоп → `price_type=range` (нижняя граница предзаполняет, финализация в вечернем дайджесте).
- **Подтверждённый список допов** (`kind=addon`), которые мастер добавляет к базе: снятие, укрепление, френч, дизайн, ремонт ногтя, наращивание ногтя.
- **Ориентиры времени допов**: снятие 10, укрепление 50, френч 20, простой дизайн 10, ремонт 1 ногтя 10, наращивание 1 ногтя 20 минут.
- **Допы за единицу несут время за единицу, не только цену** — уточнение §1. Ремонт и наращивание масштабируют время на количество, названное при записи (клиентка знает число сломанных ногтей). Ремонт масштабирует и цену (`per_unit`); наращивание — цена `on_request` (зависит от длины), но время за ноготь фиксированное. `extra_minutes` для per-unit-допа трактуется за единицу; запись захватывает количество (дефолт 1). Полноценная модель вариантов и правила сочетаний по-прежнему не строятся.
- **Разделение «клиентка знает / мастер решает»** (open by default — принимаем сразу, не гейтим): клиентка при записи сообщает старое покрытие/снятие, коррекцию, длину, френч/дизайн, число сломанных ногтей; мастер решает тип укрепления, состояние стоп и окончательную цену. Ни один из этих параметров не блокирует создание записи.
- **Персональное время под клиентку не строим сейчас**: мастер использует стандартное время; персональные ориентиры по фактической истории — позже (NAILS-004), движок ритмов заранее не строится.

Уточнено мастером (вариант «всё вместе»): базовые ориентиры «с покрытием» (130/120) — это стандартный слот, **уже включающий типичное снятие и укрепление**. Следствие для автосложения времени: снятие и укрепление **не добавляют время** поверх базы «с покрытием» (иначе двойной счёт: одно укрепление = +50 мин на каждую запись). Автосложение времени допов применяется только к действительно дополнительной работе, которой нет в стандартном слоте: **френч, дизайн, ремонт (за ноготь), наращивание (за ноготь)**. Снятие и укрепление остаются позициями каталога (цена, публичный прайс, отдельная/нестандартная запись), но во времени нейтральны к базе «с покрытием». Применимость допа к конкретной процедуре как отдельный движок не строится (deferred в §1); честный результат достигается реалистичным базовым слотом плюс переопределением времени словом.

## Инварианты (не ослабляются)

Подтверждение мастером каждой мутации (batch-занесение — одна подтверждённая мутация); снапшоты цен в записях; owner-scoping; честность ассистента (`ok=true`/`verified=true`); токен Telegram — только в profile `.env`.

## Связанное

- Hotfix `create_service` (обязательные поля без дефолтов → `invalid_arguments`) — отдельная срочная правка, не ждёт этого ADR; класс ошибки зафиксирован в engineering-principles §3.11.
- ADR-006 — та же философия дефолтов, применённая к доступности.
