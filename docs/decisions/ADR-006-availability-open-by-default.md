# ADR-006: Доступность записи открыта по умолчанию

- Статус: принято
- Дата: 17 июля 2026 года
- Связанный issue: #100

## Контекст

Обязательное предварительное «рабочее время» блокировало прямую команду мастера создать запись. Мастер называл точное время, но система требовала сначала настроить отдельный режим дня. Это лишняя пользовательская и доменная абстракция.

## Решение

Явно названное мастером время доступно по умолчанию.

Запись отклоняется только когда:

1. дата явно помечена выходным (`is_available=false`), либо
2. резерв записи пересекается с другой активной записью.

Положительные строки `availability_intervals` (`is_available=true`) задают только границы подсказок свободных окон. Они не разрешают и не запрещают явную запись.

Если положительные интервалы для даты отсутствуют, `free_slots` строит подсказки в диапазоне `10:00–23:00`. Этот диапазон также не является ограничением явной записи.

## Следствия

- `availability_unknown` перестаёт быть причиной отказа записи;
- `booking_outside_availability` не применяется к явно названному времени;
- выходной возвращает отдельную причину `booking_on_day_off`;
- overlap-проверка, buffers, owner scoping, подтверждение, audit и snapshots сохраняются;
- изменение или удаление положительного окна подсказок не конфликтует с существующей записью;
- установка выходного при наличии записи по-прежнему блокируется preview/write-проверкой;
- миграция БД не требуется.

## Не входит

- частичные отрицательные блоки: текущая схема хранит `is_available=false` как целый выходной;
- автоматический перенос или отмена записей;
- web-интерфейс и клиентский контур.

## Эталон-антипример

Нельзя требовать обязательное рабочее время перед прямой командой мастера «запиши на 08:00». Обычные часы и диапазон `10:00–23:00` служат только подсказками.
