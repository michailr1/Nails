# ADR-003: один постоянный deploy-скрипт

- Статус: принято
- Дата: 2026-07-14

## Контекст

К моменту NAILS-002E6 на деплой приложения из двух контейнеров накопилось ~3500 строк bash: отдельный runbook на каждый релиз (E4 в трёх версиях, E5, E6), хуки сохранения и восстановления образов, ручные списки ожидаемых маршрутов. Деплой E6 упал, потому что образ собрался из рабочей копии на старом коммите, а проверка маршрутов сначала опиралась на OpenAPI, сознательно отключённый в production. Каждое падение порождало новый защитный слой вместо устранения причины, и слои начали ломаться друг о друга.

## Решение

1. **Deploy-скрипт один — `ops/deploy/deploy.sh`.** Параметр релиза — точный commit SHA из `origin/main`. Новые файлы в `ops/deploy/` под конкретный релиз запрещены; скрипт меняется только через PR и CI.
2. **Тождество кода проверяется по SHA, зашитому в образ.** `docker build --build-arg GIT_SHA=<sha>` записывает коммит в `NAILS_GIT_SHA`; собранный образ проверяется до остановки runtime (изолированный `docker run --network none`), работающий контейнер — после запуска. Рукописные списки ожидаемых маршрутов в деплое не используются: ожидания дрейфуют, SHA — нет. Проверки поведения живут в тестах и CI.
3. **Сборка — только из точного дерева релиза** (`git worktree add --detach <sha>`), никогда из текущей рабочей копии.
4. **Rollback — это повторный deploy предыдущего SHA**, плюс автоматический откат образа/plugins/skills при падении в процессе. Отдельная машинерия «сохранения и восстановления образов» не используется. Следствие: миграции обязаны быть обратно-совместимыми минимум на один релиз.
5. **Runbook'и E4 (V1–V3) удалены.** E6 — последний одноразовый runbook: он завершает уже начатый релиз, после чего все деплои идут через `deploy.sh`, а E6 и его вспомогательные файлы удаляются.

## Последствия

- Класс ошибок «собралось/задеплоилось не то» устраняется физически, а не ловится проверками.
- Стоимость деплоя перестаёт расти с каждым релизом; git history заменяет версии скриптов.
- Инженерные правила, предотвращающие возврат к старому паттерну, закреплены в [engineering-principles](../operations/engineering-principles.md) и в `AGENTS.md`.
