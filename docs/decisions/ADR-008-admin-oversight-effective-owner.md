# ADR-008: Админский обзор кабинета через эффективный owner

- Статус: принято
- Дата: 2026-07-24
- Связано: ADR-004 §3.1 (owner резолвится только сервером), engineering-principles §2 (owner-scoping — священный инвариант), §3.1 (устранять класс ошибки)

## Контекст

Роль `admin` должна иметь **обзор** кабинета мастера (поддержка, диагностика, контроль). Ранее «role-aware portal» приземлился наполовину и создал прод-дефект:

- вход (`consume`, `session_state`) перевели на `web_portal_auth` с `PORTAL_ROLES = {master, admin}` — админ проходит;
- но данные (`web_read`, exports) остались master-only (`require_web_session_identity`, `role == master`);
- и все запросы скоупятся напрямую `identity.user_id`.

Итог: админ входит, сессия проходит, но каждый data-вызов даёт 401 → мгновенный откат на логин («Сессия завершилась»). Полу-состояние не является ни рабочим админ-доступом, ни чистым запретом — это дефект.

Простой «разрешить админа и в data-эндпоинтах» неверен: данные скоупятся `identity.user_id`, поэтому админ увидел бы **свой** пустой scope, а не данные мастера. Настоящего резолвинга «админ смотрит мастера X» не существует. Значит нужен явный слой, а не флип роли.

## Решение

Админский обзор строится на **эффективном owner** и **read-only** первом срезе.

### 1. Эффективный owner

Единая точка разрешения scope для всех данных:

```
resolve_effective_owner(identity, session):
  master → identity.user_id            # как сейчас, «я как мастер»
  admin  → session.target_owner_user_id # мастер, выбранный админом
           (не выбран → 409 select_master_required)
```

- Все read-эндпоинты и exports скоупятся `effective_owner_id`, а не `identity.user_id`.
- Запросы к БД используют `effective_owner_id`; audit пишет `actor_user_id = admin`, `owner = master`.

### 2. Target-owner ставится только сервером

Инвариант ADR-004 §3.1 распространяется на админа: целевой owner **никогда** не приходит из body/query data-запроса.

- `POST /web/api/admin/select-master {master_user_id}` — валидирует роль `admin`, существование и активность мастера, кладёт `target_owner_user_id` на `WebSession` вошедшего админа.
- Дальнейшие data-вызовы этой сессии резолвятся на выбранного мастера.
- Мастерская сессия шаг выбора пропускает (owner = self).

### 3. Read-only первый срез

- Админ смотрит календарь, клиенток, статистику, exports выбранного мастера.
- **Мутации остаются master-only**: для write эффективный owner обязан равняться `identity.user_id` (только мастер над собой). «Админ правит данные мастера» — отдельный поздний срез с явным consent и усиленным audit.

### 4. Видимость данных клиенток (решение 2a)

Админ в обзоре видит карточку клиентки **как мастер**, включая приватные поля (телефон, заметки о ногтях/коже, чувствительность, предпочтения). Обоснование: `admin` — доверенная операторская роль, а не недоверенная публичная аудитория ADR-004. Подотчётность обеспечивается audit-событием входа админа в scope мастера (`select-master`): видно, какой админ и когда смотрел данные какого мастера.

Замечание на будущее (multi-master): при появлении многих мастеров полная видимость означает, что админ видит чувствительные данные клиенток **всех** мастеров — тогда может понадобиться пер-мастер согласие. Для пилота (один мастер, админ = разработчик) видимость с audit-входом принята осознанно.

### 5. Регресс класса ошибки

Контрактный тест фиксирует, что **вход и данные защищены одним множеством ролей и одним owner-резолвингом**: сессия роли `admin` без выбранного мастера и сессия master получают согласованные ответы на `session`, `web_read` и exports. Ситуация «сессия ок, данные 401» становится невозможной по контракту, а не по внимательности.

### 6. Прод до готовности B

Пользователей и прод-данных нет, работой никто не затронут. Поэтому отдельная интерим-заглушка не нужна — это была бы лишняя churn (добавить роль → убрать → вернуть в B, §3.2/§3.3). Чиним сразу правильно: реализуется B. Сломанный админ-цикл до выката B никого не задевает (нет пользователей; мастер в любом случае не затронут). Конечное состояние — без полу-открытого цикла: B делает вход и данные согласованными через эффективный owner, а регресс класса ошибки закрывает «сессия ок → данные 401» по контракту.

## Модель данных

Expand-only, обратно совместимо минимум с предыдущим релизом (§3.7):

- `web_sessions.target_owner_user_id` — nullable FK → `users.id`. Заполняется только для админ-сессий через `select-master`; для мастера остаётся NULL (owner = self).
- Инвариант: непустой `target_owner_user_id` допустим только для сессии роли `admin`; целевой owner обязан быть активным мастером.

## Фронт

Тот же кабинет, без нового приложения:

- после сессии читаем роль (session уже возвращает `role`);
- `admin` без выбранного мастера → экран выбора мастера → `select-master`;
- выбранный scope → тот же (уже read-only) кабинет с плашкой «просмотр как <мастер>»;
- мастер → сразу свой кабинет.

## Инварианты (не ослабляются)

owner-scoping на каждом запросе; target-owner резолвится только сервером; identity из доверенного контекста; audit входа админа в scope мастера; write — только мастер над собственным owner.

## Рассмотрено и отклонено

- **Флип роли на data-эндпоинтах без эффективного owner** — админ увидел бы свой пустой scope, security-поверхность размывается. Отклонено.
- **Отдельное дублирующее админ-приложение** — копия кабинета ради роли, нарушает правило вычитания. Отклонено в пользу переиспользования read-only кабинета.
- **Интерим master-only заглушка перед B** — не нужна: пользователей и прод-данных нет, задевать некого, а add/remove/re-add роли — лишняя churn (§3.2/§3.3). Чиним сразу правильно через B.

## Мультимастерность

Эффективный owner + выбор мастера обобщаются напрямую: у админа picker из мастеров, резолвинг тот же. Совпадает с инвариантом клиентского контура (ADR-004): owner определяется сервером по явному выбору, никогда клиентским аргументом.
