# ADR-004: клиентский контур первой версии — детерминированный бот без LLM

- Статус: принято
- Дата: 2026-07-15

## Контекст

Доверенный контур мастера построен на LLM-ассистенте (профиль Hermes) и безопасен по конкретной причине: доступ только у двух ролей по allowlist (`master`, `admin`), противника внутри нет, а модель ограничена узкими restricted tools с подтверждениями — даже полностью «уговорённая» модель не может выйти за пределы разрешённых операций.

Клиентский контур — публичный. Аудитория недоверенная, и здесь гарантии мастер-контура не действуют. LLM-ассистент на публике создаёт класс рисков, которого нет у мастера:

- джейлбрейк и выманивание: «дай скидку 50%», чужой слот, сведения о других клиентах;
- обязательства из обещаний бота: обещание, сгенерированное моделью, может связывать бизнес юридически (прецедент: суд обязал авиакомпанию исполнить обещание её чат-бота);
- prompt injection и злоупотребление: посторонний тратит токены и время на «поговорить с ботом»;
- необходимость постоянной гонки щитов против новых обходов.

При этом реальные потребности клиента узкие и не требуют диалога: увидеть свободные окна, попроситься на запись, перенести или отменить свою запись, получить напоминание. Это кнопочные сценарии, а не разговор.

## Решение

**Клиентский контур первой версии — детерминированный Telegram-бот без LLM.**

1. Интерфейс — inline-кнопки и фиксированные шаблоны текста; свободный разговор не поддерживается.
2. Прямые вызовы того же Booking API с ролью `client`: клиент видит и меняет только свои записи.
3. Запись со стороны клиента — это **заявка**, которую подтверждает мастер в доверенном контуре; автоподтверждения нет.
4. Цены, услуги и окна берутся из каталога и расписания шаблоном — бот их не формулирует и не обещает.
5. Свободный текст от клиента не интерпретируется: бот отвечает «передам мастеру» и форвардит сообщение в доверенный контур. Человек в петле — корректный обработчик нестандартных случаев при масштабе одного мастера.
6. Клиентский бот имеет **отдельную личность**, отличную от ассистента мастера: тёплый тон «своего человека» из доверенного контура не переносится в бот, говорящий с посторонними.

## Последствия

- Джейлбрейк, выманивание и prompt injection устраняются как класс: нет модели, промпта и разговорной поверхности.
- Бот не может пообещать скидку или чужой слот — данные только из каталога и API.
- Ответы мгновенные (нет задержки инференции модели, ср. Issue #69), стоимость эксплуатации близка к нулю.
- Booking API получает роль `client` со строго ограниченными операциями (свои записи, заявки) — отдельно от `master`/`admin`.
- Соразмерность (engineering-principles §1): для клиента, который обращается редко и кнопочно, LLM-диалог избыточен; для мастера с десятками составных запросов в день — оправдан.

## Границы и будущее

- Разговорный FAQ-режим для клиентов (например, вопросы об услугах) — **опция на потом**, а не условие запуска. Если понадобится, он делается отдельным изолированным read-only компонентом: ответы только из каталога, отдельная личность, жёсткие фильтры и rate limit. Это не «публичный Hermes мастер-контура».
- Клиентский контур не начинается, пока мастер не живёт на системе (engineering-principles §5).

## Рассмотрено и отклонено

- **Публичный профиль Hermes для клиентов.** Переносит LLM-поверхность и весь класс рисков недоверенной аудитории в продукт ради сценариев, которые решаются кнопками. Отклонено как несоразмерное и небезопасное.
