# ADR-005: веб-интерфейс мастера с авторизацией через Telegram

- Статус: принято
- Дата: 2026-07-16

## Контекст

Мастеру нужен визуальный интерфейс: дашборды (день/неделя, занятость, выручка), карточки клиенток, каталог услуг — с возможностью не только смотреть, но и **редактировать** записи, услуги и карточки. Telegram-диалог удобен для быстрых действий, но обзор недели и правку мышкой веб делает нагляднее.

Это **первый публичный компонент проекта с правом записи**. До сих пор единственная внешняя поверхность — Telegram-бот с allowlist по ID; публичного веб-доступа не было. Появление веб-приложения в интернете, меняющего данные, — значимый рост attack surface, поэтому решение фиксируется отдельно.

## Решение

1. **Веб — тонкий слой над существующим Booking API.** Все мутации (записи, услуги, клиентки, доступность) выполняются через те же owner-scoped операции, что использует Нэйли: проверка пересечений, снимки цен, приватность `internal_alias`/`internal_notes`, аудит. Веб **не обращается к БД напрямую и не дублирует бизнес-логику**.
2. **Две раздельные поверхности.** Внутренний API (`127.0.0.1:8210`, доверяющий заголовку идентичности от plugin) остаётся закрытым на loopback и **наружу не выставляется**. Публичный веб — отдельный сервис/роутер с собственной аутентификацией на отдельном нестандартном порту.
3. **Авторизация — одноразовый код через Telegram.** Вход разрешён только для Telegram ID из allowlist (`master`/`admin`). Код: не короче 6 цифр, TTL ≈ 5 минут, одноразовый, ограниченное число попыток ввода и rate-limit запросов кода. После верификации — серверная сессия: `secure`, `httponly`, `samesite` cookie, ограниченный срок, ротация, явный logout.
4. **TLS обязателен.** Публичная поверхность с кодами и сессиями работает только по HTTPS с валидным сертификатом. Нестандартный порт допустим (например `8443`); порт «не 443» сам по себе не является защитой.
5. **CSRF-защита** на всех мутирующих запросах; аудит веб-действий; явная проверка, что внутренние поля не попадают туда, где им не место.
6. **Реальные данные — только после NAILS-002F** (подтверждённый restore-тест). Прототип на синтетических данных допустим раньше.

## Последствия

- Мастер получает наглядный интерфейс с правкой; бизнес-правила не дублируются — переиспользуется Booking API.
- Появляется публичный attack surface: требуются крепкая auth (OTP + сессии), TLS, rate-limit, CSRF и периодический security-review именно этого компонента.
- Веб и Telegram — два интерфейса к одним данным; оба идут через Booking API, поэтому остаются согласованными. Модель подтверждения различается корректно: в Telegram Нэйли подтверждает перед мутацией, в вебе подтверждением служит явное действие пользователя (кнопка «Сохранить»).

## Границы

- Это интерфейс **мастера** (роли `master`/`admin`). Он **не** предназначен для клиенток: клиентский контур строится по ADR-004 (детерминированно, без LLM, отдельная изоляция), а не как «админка мастера с расширенным доступом».
- Развёртывание — через тот же `ops/deploy/deploy.sh` (ADR-003), веб как дополнительный сервис Compose; отдельные релизные runbook'и не создаются.

## Рассмотрено и отклонено

- **Прямой доступ веба к БД в обход Booking API** — раздваивает бизнес-правила, отклонено.
- **Выставление внутреннего API наружу** — заголовок идентичности доверенный только на loopback; публично его подделают. Отклонено.
- **Публичный доступ без TLS или без ограничения allowlist** — отклонено как небезопасное.
- **«Просто открыть порт наружу без auth»** — нестандартный порт не защита; отклонено.
