# ADR-002: стек, Telegram-контур и развёртывание

- Статус: принято
- Дата: 2026-07-12

## Контекст

На VPS уже развёрнут Hermes и используется его встроенная интеграция с Telegram. Добавление отдельного Telegram-фреймворка создало бы лишний транспортный слой, дублирование сессий и allowlist, а также риск конкуренции за обновления одного bot token.

Проекту при этом всё равно нужен отдельный backend, который хранит данные и применяет бизнес-правила независимо от модели.

## Решение

### Telegram и Hermes

- Telegram-транспортом служит встроенный Hermes Telegram Gateway;
- отдельный aiogram-процесс не используется;
- для Nails создаётся отдельный Telegram-бот и отдельный bot token;
- создаётся отдельный профиль Hermes с техническим именем `nails`;
- профиль использует отдельные сессии, память, системные инструкции и allowlist;
- подключаются skills `nails-onboarding` и `nails-scheduling`;
- Telegram ID берётся из доверенного gateway context, а не из текста сообщения или аргумента модели;
- роль `master` или `admin` окончательно определяется и проверяется Booking API.

### Backend

Принят стек:

- Python;
- FastAPI;
- PostgreSQL;
- SQLAlchemy;
- Alembic;
- Pydantic Settings;
- pytest;
- Ruff.

Booking API является единственной точкой записи и проверки бизнес-правил. Hermes не выполняет прямые SQL-запросы и не изменяет данные вне разрешённых API-функций.

### Развёртывание

Nails разворачивается на том же VPS, где уже работает Hermes, но в отдельном изолированном контуре:

```text
Hermes
└── profile: nails
    ├── отдельный Telegram bot token
    ├── отдельный allowlist
    ├── отдельные сессии и память
    ├── nails-onboarding
    └── nails-scheduling

/opt/nails/
├── Booking API
├── PostgreSQL или отдельная БД и пользователь
├── worker
└── backup tooling
```

Разрешается использовать общий экземпляр PostgreSQL только при наличии отдельной базы или схемы, отдельного пользователя и минимальных прав.

Резервные копии должны сохраняться вне рабочей БД и вне того же единственного диска VPS.

## Ограничения профиля Hermes

Профилю `nails` не выдаются:

- shell или terminal;
- SSH;
- произвольный файловый доступ;
- прямой SQL;
- универсальный HTTP-клиент;
- credentials Google Calendar;
- инструменты и секреты других профилей Hermes.

Профиль получает только строго определённые инструменты Booking API.

## Кнопки Telegram

На первом рабочем этапе допускается подтверждение обычным текстом или простыми вариантами ответа, если Hermes Gateway не предоставляет стабильный универсальный интерфейс для пользовательских inline-кнопок.

Отдельный aiogram-слой рассматривается только в будущем, если штатный Telegram Gateway Hermes принципиально не сможет реализовать необходимый интерфейс. Это потребует отдельного архитектурного решения.

## Последствия

Преимущества:

- отсутствует дублирование Telegram-транспорта;
- используется уже работающий контур Hermes;
- меньше компонентов для развёртывания и сопровождения;
- отдельный специализированный бот не смешивается с основным Hermes-ботом;
- бизнес-данные и правила остаются в контролируемом Booking API.

Ограничения:

- интерфейс Telegram зависит от возможностей Hermes Gateway;
- кастомные callback-кнопки должны быть отдельно проверены до реализации;
- состояние onboarding нельзя хранить только в памяти Hermes — оно сохраняется в PostgreSQL.
