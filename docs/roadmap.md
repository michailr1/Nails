# Roadmap

План построен вертикальными срезами: каждый этап заканчивается работающей функцией, которую мастер может проверить в живом диалоге. Цель — получить обратную связь как можно раньше и не делать Google Calendar критической зависимостью.

Технологический стек, использование Hermes Telegram Gateway и развёртывание на существующем VPS зафиксированы в [`ADR-002`](decisions/ADR-002-technology-and-deployment.md).

## NAILS-001 — фиксация процессов и правил

До начала реализации необходимо согласовать:

- роли и полномочия;
- каталог услуг, публичные описания и цены;
- стандартные длительности и буферы;
- правила персональной длительности;
- рабочий график и исключения;
- обязательный локальный часовой пояс в формате IANA;
- статусы записей, включая `completed`, `cancelled`, `no_show`;
- правила снимка стоимости и расчёта выручки;
- подтверждение опасных операций;
- минимальную и расширенную карточку клиента;
- разделение публичных и внутренних полей;
- формат компактного просмотра дня;
- политику backup и проверки восстановления;
- политику feedback-логов;
- минимальные метрики пилота.

Общие правила считаются достаточными для начала разработки. Конкретные услуги, цены, длительности и график мастера собираются через onboarding-интервью NAILS-002.

## NAILS-002 — сквозной happy path

Минимальный, но живой контур: мастер знакомится с ботом, проходит интервью, создаёт простую карточку, спрашивает окна, записывает клиента и смотрит день.

Telegram-транспорт реализует встроенный Hermes Telegram Gateway. Отдельный aiogram-процесс не используется.

### NAILS-002A — каркас backend

- Python + FastAPI;
- SQLAlchemy + Alembic;
- PostgreSQL;
- Pydantic Settings;
- pytest и Ruff;
- обязательный `APP_TIMEZONE` без автоматического значения по умолчанию;
- миграции для `users`, `services`, `clients`, `bookings`, `schedule_rules`, `audit_events`, onboarding-состояния и черновиков;
- CI: миграции на чистой БД, линтер и тесты;
- deployment-конфигурация для отдельного `/opt/nails/` на существующем VPS.

### NAILS-002B — onboarding API

- `start_onboarding`;
- `get_onboarding_state`;
- сохранение черновика графика;
- сохранение черновика услуг, цен и длительностей;
- сохранение черновика буферов;
- сохранение черновика записей наперёд;
- подтверждение или исправление каждого блока;
- `pause_onboarding` и `resume_onboarding`;
- `complete_onboarding`;
- после перезапуска Hermes и backend интервью продолжается с подтверждённого шага.

Черновые данные не участвуют в рабочем расписании до подтверждения блока.

### NAILS-002C — Hermes Telegram Gateway и профиль

- отдельный Telegram-бот и token;
- отдельный профиль Hermes `nails`;
- отдельные allowlist, сессии, память и инструкции;
- доступ только для `master` и `admin`;
- Telegram ID передаётся через trusted gateway context;
- роль проверяется Booking API;
- запрещённому пользователю не раскрывается информация о системе;
- подтверждения могут работать текстом, если универсальные inline-кнопки Hermes пока недоступны.

### NAILS-002D — skill `nails-onboarding`

- знакомство и объяснение только реально включённых функций;
- ответы на вопросы мастера;
- неизвестные вопросы фиксируются для администратора и не блокируют интервью;
- интервью: график → услуги, цены и длительности → буферы → записи наперёд;
- короткие возобновляемые сессии;
- сводка и подтверждение после каждого блока;
- capability flags, чтобы Hermes не обещал нереализованные функции;
- кнопка или текстовый эквивалент «не то» с защищённым логом.

### NAILS-002E — первый scheduling happy path

- простое создание клиента с обязательным `public_name` и необязательным контактом;
- точный поиск клиента по нормализованному публичному имени;
- поиск свободных окон;
- создание записи со снимком стоимости и валюты;
- просмотр расписания дня в компактном формате;
- skill `nails-scheduling` получает только соответствующие функции Booking API;
- сразу после интервью должны работать запросы «что у меня в четверг?», «какие окна завтра?» и создание тестовой записи.

### NAILS-002F — backup и восстановление

- автоматические бэкапы PostgreSQL;
- копия вне рабочей БД и вне единственного диска VPS;
- журнал успешности backup;
- тестовое восстановление в отдельную БД;
- документированный результат восстановления до пилота на реальных данных.

### Контрольная точка NAILS-002

- весь onboarding сначала проходит роль `admin`;
- после интервью первый запрос по расписанию возвращает данные, только что внесённые через диалог;
- профиль `nails` не имеет shell, SSH, прямого SQL, универсального HTTP и секретов других профилей;
- запасной фотоимпорт записной книжки не блокирует happy path и реализуется отдельно после основной проверки.

## NAILS-003 — управление записями

- переносы и отмены;
- блокировки времени: интервал, день, диапазон дат;
- подтверждение опасных операций;
- идемпотентность повторных запросов;
- задержка процедуры и отображение конфликтов без автоматического сдвига дня;
- просмотр недели и произвольного периода;
- расширение skill `nails-scheduling` и функций Booking API под операции этапа;
- срок хранения и ручное удаление feedback-событий.

## NAILS-004 — расширенная клиентская база

- внутренние алиасы и заметки;
- поиск точных и похожих совпадений;
- защита от опечаток и предупреждение о возможном дубле;
- персональные корректировки длительности и цены (`client_service_overrides`);
- расширение skill `nails-scheduling` и функций Booking API;
- полноценное объединение дублей не входит в MVP.

## NAILS-005 — повседневные сценарии

- «записать как обычно»: повтор последней завершённой услуги с персональной длительностью;
- статус `no_show`, число неявок вычисляется по истории записей;
- утренняя сводка расписания дня в заданное время;
- завершение визита и фиксация фактической итоговой цены;
- итоги дня и недели по снимкам стоимости;
- расширение skill `nails-scheduling` и функций Booking API;
- минимальные агрегированные метрики пилота.

## NAILS-006 — Google Calendar

- отдельный календарь;
- безопасное хранение credentials;
- таблица или очередь `calendar_sync_jobs`;
- односторонний экспорт подтверждённых изменений из БД;
- повторы с backoff;
- уведомление администратора после исчерпания повторов;
- фоновая сверка и восстановление рассинхронизации;
- недоступность календаря не блокирует работу с записями.

## NAILS-007 — тестирование

- двойная запись;
- перенос на занятый интервал;
- неоднозначные имена;
- одинаковые публичные имена у разных карточек;
- поиск по внутреннему алиасу;
- защита внутренних обозначений;
- повторное подтверждение операции;
- ролевые ограничения;
- запись, заканчивающаяся ровно в конец рабочего дня;
- блокировка, переходящая через полночь;
- переходы летнего и зимнего времени;
- изменение цены услуги не меняет историческую выручку;
- `no_show` не считается обычной отменой;
- задержка не сдвигает следующие записи автоматически;
- feedback-события удаляются по сроку хранения;
- сбои Google Calendar не блокируют бизнес-операции;
- backup действительно восстанавливается.

## NAILS-008 — пилот

- параллельное ведение старого и нового расписания;
- сверка записей;
- разбор feedback-логов и нераспознанных запросов;
- контроль минимальных метрик пилота;
- корректировка услуг и длительностей;
- проверка backup и восстановления;
- переход на основное использование только после подтверждения стабильности.

## Следующий этап, не входящий в MVP

- отдельный публичный бот;
- самостоятельная запись клиентов;
- консультации по услугам и стоимости;
- обязательное подтверждение мастером;
- отдельная изоляция клиентского агента;
- security monitoring и защита от злоупотреблений;
- полноценное объединение дублей карточек как отдельная административная функция.
