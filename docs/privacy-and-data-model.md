# Приватность и модель данных

## 1. Классы данных

### Публично допустимые поля

Поля, которые в будущем могут использоваться в сообщении клиенту:

- `public_name`;
- публичное название услуги;
- дата и время записи;
- публичная стоимость;
- публичные правила посещения.

### Внутренние поля

Доступны только ролям `master` и `admin`:

- `internal_alias`;
- `internal_notes`;
- служебные теги;
- история изменений;
- технические идентификаторы.

Внутренние значения не передаются в клиентский контур.

## 2. Предварительная модель

### `users`

- `id`;
- `telegram_user_id`;
- `role`;
- `is_active`;
- timestamps.

### `clients`

- `id`;
- `public_name`;
- `normalized_public_name`;
- контактные поля;
- `profile_status`;
- timestamps;
- `archived_at`.

### `client_internal_aliases`

- `id`;
- `client_id`;
- `alias`;
- `normalized_alias`;
- `created_by`;
- `created_at`.

### `client_internal_notes`

- `id`;
- `client_id`;
- `note`;
- `visibility = master_admin_only`;
- `created_by`;
- timestamps.

### `services`

- `id`;
- публичное название и описание;
- стоимость или правило расчёта;
- длительность;
- буферы до и после;
- `is_active`.

### `client_service_overrides`

Персональные корректировки длительности для пары «клиент + услуга»:

- `id`;
- `client_id`;
- `service_id`;
- корректировка длительности;
- `created_by`;
- timestamps.

Используется при расчёте окон вместо чтения длительности из текста заметок.

### `bookings`

- `id`;
- `client_id`;
- `service_id`;
- начало и окончание;
- статус (включая `no_show` как отдельный статус, отличный от отмены);
- `calendar_event_id`;
- `idempotency_key`;
- timestamps.

### `schedule_rules` и `schedule_exceptions`

Хранят обычный рабочий график и разовые изменения.

### `audit_events`

Хранят актёра, действие, объект, безопасное описание изменения, request ID и время. Секреты и полные чувствительные payload в аудит не записываются.

### `feedback_events`

Лог разбора качества ответов:

- `id`;
- контекст диалога на момент нажатия 👎 или нераспознанного запроса;
- тип (`thumbs_down` / `unrecognized`);
- `created_at`.

Содержит реальные тексты диалогов, поэтому относится к внутренним данным: хранится только в защищённом контуре и никогда не попадает в GitHub.

## 3. GitHub

Разрешены только синтетические данные, например:

```text
public_name = "Тестовый клиент A"
phone = "+70000000001"
telegram_user_id = 100000001
```

Запрещено сохранять реальные данные даже в issues, pull requests, Actions logs и скриншотах.

## 4. Исходящие сообщения

Будущий сервис отправки сообщений должен получать только whitelisted public-поля. Перед отправкой выполняется проверка, что текст не содержит значения внутренних алиасов и заметок.
