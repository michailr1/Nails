# Модуль `nails-onboarding` — паспорт проприетарного модуля

- Версия модуля: **0.1.0**
- Статус: реализован (NAILS-002C), production-установка выполняется отдельным шагом
- Расположение в репозитории: `hermes/plugins/nails_onboarding/`
- Принадлежность: **проприетарный внутренний модуль проекта Nails**. Использование, копирование и распространение за пределами проекта не допускаются. Модуль не предназначен для установки в чужие профили Hermes.

Этот документ описывает модуль обычным языком: зачем он нужен, из чего состоит и что делает каждая его функция. Технические требования безопасности и приёмки вынесены в [hermes-onboarding-plugin.md](../hermes-onboarding-plugin.md).

## 1. Зачем нужен модуль

Ассистент Smart Nails общается с мастером в Telegram, но сам по себе он не умеет ничего сохранять — все данные живут в Booking API и PostgreSQL. Модуль `nails-onboarding` — это **единственный мост** между диалогом и сервером на этапе первичной настройки (онбординга).

Почему мост сделан отдельным узким модулем, а не обычным HTTP-доступом:

- ассистент физически не может отправить запрос «куда-то ещё» — адрес сервера зашит в код;
- ассистент не знает и не может узнать ключ доступа — модуль сам берёт его из защищённого окружения;
- ассистент не может выдать себя за другого пользователя — личность берётся только из доверенного контекста Telegram-шлюза, а попытка передать чужой ID в аргументах отклоняется до любого сетевого вызова;
- ассистенту доступны ровно семь операций онбординга — ни SQL, ни файлов, ни произвольных команд.

Проще говоря: даже если модель «захочет» сделать лишнее, у неё нет для этого рук.

## 2. Из чего состоит модуль

| Файл | Что это |
|---|---|
| `plugin.yaml` | Паспорт модуля для Hermes: имя, версия, требуемая секретная настройка `NAILS_INTERNAL_API_KEY`, список предоставляемых инструментов |
| `__init__.py` | Точка подключения: функция `register()` регистрирует в Hermes один инструмент `nails_onboarding` |
| `schemas.py` | Описание инструмента для модели: какие аргументы разрешены (`action`, `section`, `payload`) и что они значат |
| `tools.py` | Вся логика: проверка аргументов, получение доверенной личности, вызов Booking API, безопасная обработка ошибок |
| `tests/test_plugin.py` | 12 автотестов, включая проверки защиты от подмены личности и утечки секретов |

## 3. Операции, доступные ассистенту

Ассистент вызывает один инструмент — `nails_onboarding` — и указывает действие (`action`). Все действия работают только для того пользователя Telegram, который сейчас пишет боту.

### `start` — начать настройку

Создаёт для пользователя новое интервью настройки, если его ещё не было. Если интервью уже идёт — просто возвращает его текущее состояние (повторный вызов безопасен). Если интервью на паузе, подскажет использовать `resume`; если уже завершено — сообщит об этом.

### `get_state` — узнать, где мы остановились

Возвращает полную картину: статус интервью (идёт / на паузе / завершено), текущий шаг и состояние каждого блока — что сохранено черновиком, что подтверждено. Используется в начале разговора, чтобы продолжить с нужного места, и после перезапуска бота.

### `save_section` — сохранить черновик блока

Сохраняет данные одного блока (`section`) с содержимым (`payload`) — например, рабочий график, о котором мастер только что рассказала. Сохранённое — это **черновик**: его можно менять сколько угодно, каждое изменение получает новый номер ревизии. Если прислать те же данные повторно, ничего не изменится (защита от дублей). Сервер проверяет данные по строгой схеме и вернёт понятный список ошибок, если что-то не так (например, у рабочего дня не указано время).

### `confirm_section` — подтвердить блок

Фиксирует черновик блока как подтверждённый. Вызывается только после того, как мастер увидела сводку и явно согласилась. Правила:

- блоки подтверждаются по порядку: график → услуги → буферы → записи;
- график подтверждается только целиком — все семь дней недели;
- буферы и записи могут ссылаться только на уже подтверждённые услуги;
- если переподтвердить ранний блок (например, изменить услуги), подтверждения зависимых блоков автоматически снимаются — их нужно пройти заново;
- повторное подтверждение того же самого — безопасно, ничего не ломает.

### `pause` — приостановить интервью

Ставит интервью на паузу: всё сохранённое остаётся на сервере, продолжить можно в любой момент. Повторная пауза безопасна.

### `resume` — продолжить интервью

Снимает паузу и возвращает состояние, чтобы ассистент продолжил с последнего неподтверждённого шага.

### `complete` — завершить настройку

Отмечает интервью завершённым. Сработает только когда **все** блоки подтверждены в актуальных версиях — иначе вернёт список того, что осталось. Повторный вызов безопасен.

Важно: завершение онбординга означает «данные собраны и подтверждены». Перенос их в рабочее расписание (материализация) — отдельный следующий этап (NAILS-002E), и до него ассистент обязан говорить об этом честно.

## 4. Блоки настройки (`section`)

| Блок | Что в нём |
|---|---|
| `schedule` | Рабочая неделя: по каждому дню — рабочий ли он и часы работы |
| `services` | Услуги: название, описание, цена, валюта, длительность в минутах |
| `buffers` | Технические зазоры до и после конкретных услуг (например, время на уборку) |
| `bookings` | Существующие записи из записной книжки: клиентка, телефон (необязательно), услуга, дата и время |

## 5. Что модуль возвращает

Всегда JSON-строка одного из двух видов.

Успех:

```json
{"ok": true, "action": "save_section", "result": { ...состояние онбординга... }}
```

Ошибка:

```json
{"ok": false, "error": {"code": "...", "message": "понятное сообщение"}}
```

### Коды ошибок модуля

| Код | Что означает | Что делать ассистенту |
|---|---|---|
| `invalid_arguments` | Аргументы вызова не прошли проверку (лишние поля, неизвестное действие, section там, где не нужен) | Исправить вызов; пользователю это не показывается как его ошибка |
| `trusted_context_required` | Вызов не из доверенной Telegram-сессии | Объяснить, что настройка доступна только из Telegram |
| `plugin_not_configured` | На сервере не задан ключ доступа к API | Сказать, что настройка временно недоступна, сообщить администратору |
| `access_denied` | Пользователю нельзя пользоваться онбордингом (нет в списке, отключён — модуль намеренно не уточняет) | Нейтрально сказать, что аккаунту сейчас недоступна настройка |
| `service_unavailable` | Booking API не отвечает или временная ошибка (после одной повторной попытки) | Сказать, что сохранение временно недоступно, предложить повторить позже |
| `invalid_backend_response` | Сервер ответил в неожиданном формате | Как `service_unavailable` |
| `internal_tool_error` | Непредвиденный сбой самого модуля (записан в лог) | Как `service_unavailable` |

### Коды ошибок, приходящие от Booking API

Модуль пропускает наружу только безопасный код и детали:

| Код | Что означает |
|---|---|
| `onboarding_not_started` | Интервью ещё не начиналось — сначала `start` |
| `onboarding_paused_use_resume` | Интервью на паузе — продолжать через `resume` |
| `onboarding_already_completed` | Интервью уже завершено |
| `onboarding_not_in_progress` | Действие возможно только в идущем интервью |
| `section_draft_missing` | Блок нельзя подтвердить — черновик ещё не сохранён |
| `prior_sections_not_confirmed` | Сначала нужно подтвердить предыдущие блоки (в деталях — какие) |
| `schedule_requires_all_weekdays` | В графике описаны не все семь дней (в деталях — какие отсутствуют) |
| `services_not_confirmed` | Буферы и записи требуют подтверждённого блока услуг |
| `unknown_service_reference` | Названа услуга, которой нет среди подтверждённых (в деталях — какие) |
| `invalid_onboarding_payload` | Данные блока не прошли проверку (в деталях — поля и причины) |
| `onboarding_sections_not_confirmed` | Завершить нельзя — есть неподтверждённые блоки (в деталях — какие) |

## 6. Внутренние функции (для разработчиков)

Что делает каждая функция `tools.py`:

| Функция | Назначение простыми словами |
|---|---|
| `nails_onboarding(args)` | Главный обработчик: проверяет аргументы, получает личность и ключ, вызывает сервер, возвращает безопасный ответ. Любое исключение превращает в безопасную ошибку — модуль никогда не «взрывается» в диалог |
| `_validate_args(args)` | Пропускает только `action`, `section`, `payload` в разрешённых сочетаниях; всё лишнее (включая попытки подсунуть чужой ID) отклоняет до сети |
| `_trusted_telegram_user_id()` | Берёт платформу и ID пользователя из доверенного контекста шлюза Hermes; если это не Telegram или ID некорректен — отказ |
| `_api_key()` | Читает ключ `NAILS_INTERNAL_API_KEY` из окружения; короче 32 символов или отсутствует — отказ без сетевого вызова |
| `_request_spec(action, …)` | Сопоставляет действие с конкретным HTTP-методом и путём API — только семь фиксированных маршрутов |
| `_call_backend(…)` | Выполняет запрос к `127.0.0.1:8210` с ключом, ID пользователя и уникальным request ID; делает не более одной повторной попытки и только при сетевых сбоях или 502/503/504; превращает ответы сервера в безопасный результат (401 и 403 — в одинаковый `access_denied`) |
| `_safe_backend_detail(response)` | Достаёт из ошибки сервера только код и безопасные детали; всё неожиданное заменяет на общий `backend_error` |
| `_http_request(…)` | Единственная точка реального HTTP-вызова (таймаут 5 секунд, редиректы запрещены); в тестах подменяется |
| `_json_response(payload)` | Сериализует ответ в компактный JSON с поддержкой кириллицы |
| `_get_session_env(name)` | Тонкая обёртка над `gateway.session_context` Hermes — вынесена отдельно, чтобы тесты могли подменять контекст |
| `register(ctx)` (`__init__.py`) | Регистрирует в Hermes ровно один инструмент `nails_onboarding` в одноимённом toolset |

## 7. Установка и настройка

- Модуль устанавливается **только** в профиль `nails`: `/root/.hermes/profiles/nails/plugins/nails_onboarding`.
- В окружение профиля передаётся `NAILS_INTERNAL_API_KEY` (тот же ключ, что `INTERNAL_API_KEY` у Booking API, минимум 32 символа). Значение не выводится на экран и не попадает в логи и ответы.
- В whitelist инструментов Telegram добавляется только toolset `nails_onboarding`.
- Booking API должен быть доступен на `127.0.0.1:8210` (значение зашито в код намеренно и меняется только правкой кода через ревью).

## 8. Чего модуль не делает

- не создаёт пользователей и не управляет allowlist;
- не переносит подтверждённые данные в рабочее расписание (это NAILS-002E);
- не ищет свободные окна и не создаёт рабочие записи;
- не даёт произвольный HTTP, SQL или доступ к файлам;
- не работает вне Telegram-сессии.
