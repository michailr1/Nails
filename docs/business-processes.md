# Бизнес-процессы первого этапа

## 1. Поиск свободного времени

Мастер задаёт услугу и период обычным сообщением. Агент определяет необходимую длительность, Booking API учитывает рабочий график, существующие записи, блокировки, буферы и персональную корректировку длительности, после чего возвращает подходящие окна.

## 2. Создание записи

Обязательные данные:

- клиент;
- услуга;
- дата и время;
- рассчитанная длительность;
- снимок стоимости и валюта.

Перед записью Booking API повторно проверяет отсутствие конфликта. Неоднозначные операции подтверждаются кнопкой.

В первом рабочем срезе допускается быстрое создание минимальной карточки клиента:

- обязательное публичное имя;
- контакт необязателен;
- выполняется точный поиск по нормализованному имени;
- расширенный поиск опечаток, алиасы и заметки добавляются позднее.

Цена сохраняется в записи как снимок. Последующее изменение каталога услуг не меняет стоимость старых записей.

## 3. Диалоговое формирование базы клиентов

Расширенная база формируется в ходе обычной работы.

При упоминании клиента агент:

1. нормализует введённое обозначение;
2. ищет точные совпадения;
3. ищет совпадения по внутренним алиасам;
4. ищет похожие варианты и возможные опечатки;
5. показывает кандидатов без раскрытия лишних данных;
6. просит подтвердить существующую карточку либо создание новой;
7. уточняет публичное имя;
8. отдельно уточняет внутреннее обозначение мастера;
9. запрашивает минимальный контакт, позволяя временно пропустить его;
10. создаёт карточку и продолжает исходную операцию.

Автоматически объединять похожие карточки запрещено.

В MVP агент только предупреждает о возможном дубле. Полноценное объединение карточек переносится на отдельную административную функцию после пилота.

## 4. Публичное имя и внутреннее обозначение

Каждая карточка разделяет:

- `public_name` — допустимое имя для будущего обращения к клиенту;
- `internal_alias` — привычное обозначение мастера для поиска;
- `internal_notes` — закрытые рабочие заметки.

Внутренние обозначения и заметки не используются в клиентских сообщениях, публичных шаблонах или интеграциях.

Пример синтетических данных:

```text
public_name: Клиент A
internal_alias: Клиент A — увеличенное время
internal_notes: Предпочитает вечерние записи
```

Правило дополнительного времени хранится не в заметке, а в `client_service_overrides`.

## 5. Перенос записи

Агент находит запись, показывает её параметры, предлагает или проверяет новое время и запрашивает подтверждение.

После подтверждения:

1. Booking API меняет запись в PostgreSQL;
2. сохраняет аудит;
3. создаёт задание односторонней синхронизации календаря, если интеграция подключена;
4. возвращает успешный результат мастеру.

Сбой Google Calendar не отменяет перенос в основной системе. Экспорт повторяется фоновым worker.

## 6. Отмена записи

Агент показывает найденную запись и запрашивает подтверждение. После подтверждения:

- запись получает статус отмены;
- действие фиксируется в аудите;
- создаётся задание на удаление или пометку события календаря;
- ошибка календаря не восстанавливает отменённую запись в PostgreSQL.

## 7. Блокировка времени

Мастер может закрыть интервал, день или диапазон дат. Блокировки участвуют в расчёте доступности наравне с записями.

Блокировка, переходящая через полночь, явно разбивается системой по календарным дням в часовом поясе мастера либо хранится как timezone-aware интервал с корректной проверкой пересечений.

## 8. Просмотр расписания

Поддерживаются запросы:

- на день;
- на неделю;
- на выбранный период;
- по клиенту;
- по услуге;
- список свободных окон.

Компактные форматы дня и недели являются отдельной UX-функцией и проверяются в пилоте.

## 9. Повторная запись «как обычно»

Мастер может записать постоянного клиента одной фразой, например «как обычно, через три недели».

Агент:

1. находит последнюю подходящую завершённую запись;
2. берёт услугу и персональную длительность из структурированных данных;
3. показывает найденные параметры;
4. предлагает свободные окна;
5. создаёт новую запись после подтверждения.

## 10. Персональная длительность

Если процедура у конкретного клиента регулярно занимает больше или меньше стандартного времени, корректировка хранится в `client_service_overrides` и автоматически участвует в расчёте окон.

Агент не извлекает длительность из свободного текста внутренних заметок.

## 11. Неявка (no-show)

Если клиент не пришёл, запись получает статус `no_show` — это отдельный статус, а не отмена.

Количество неявок вычисляется по записям со статусом `no_show`, а не хранится отдельным изменяемым счётчиком в карточке клиента.

## 12. Задержка или увеличение фактического времени

Мастер может сообщить:

> Процедура задерживается на 20 минут.

Система:

- сохраняет ожидаемое или фактическое время окончания;
- показывает конфликт со следующей записью;
- предлагает доступные действия;
- не переносит следующие записи автоматически.

Изменение последующих записей выполняется только по решению мастера.

## 13. Утренняя сводка

В заданное мастером время бот сам присылает расписание дня в компактном формате. Функция отключаемая.

## 14. Итоги дня и недели

Выручка рассчитывается по снимку стоимости, сохранённому в каждой записи, а не по текущей цене услуги в каталоге.

Правила расчёта:

- учитываются записи со статусом `completed`;
- отменённые записи не входят в выручку;
- `no_show` входит только в размере фактически удержанной предоплаты или штрафа, если такая функция появится;
- ручная корректировка итоговой цены фиксируется в аудите;
- отчёт показывает валюту и не смешивает разные валюты в одну сумму.

## 15. Обратная связь о качестве ответов

Под ответами бота доступна кнопка «не то» (👎). Нажатие сохраняет ограниченный контекст диалога без прерывания работы.

Правила:

- сохраняются только последние сообщения, необходимые для разбора;
- секреты удаляются, контакты по возможности маскируются;
- доступ имеет только `admin`;
- срок хранения настраивается, начальное значение — 30 дней;
- событие можно удалить вручную;
- нераспознанные запросы подчиняются тем же правилам.

## 16. Минимальные метрики пилота

В пилоте отслеживаются:

- количество успешно созданных записей;
- исправленные и отменённые операции;
- повторные уточнения;
- нажатия 👎 и нераспознанные запросы;
- расхождения со старым расписанием;
- ошибки резервного копирования;
- ошибки календарной синхронизации после её подключения.

## 17. Административные процессы

Администратор управляет:

- доступом пользователей;
- услугами, стоимостью и длительностью;
- рабочим графиком и буферами;
- техническими настройками;
- аудитом и диагностикой;
- просмотром и удалением feedback-событий;
- ручным исправлением ошибочных карточек.

Полноценное объединение дублей не входит в MVP.
