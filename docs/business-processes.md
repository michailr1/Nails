# Бизнес-процессы первого этапа

## 1. Поиск свободного времени

Мастер задаёт услугу и период обычным сообщением. Агент определяет необходимую длительность, Booking API учитывает рабочий график, существующие записи, блокировки и буферы, после чего возвращает подходящие окна.

## 2. Создание записи

Обязательные данные:

- клиент;
- услуга;
- дата и время;
- рассчитанная длительность.

Перед записью Booking API повторно проверяет отсутствие конфликта. Неоднозначные операции подтверждаются кнопкой.

## 3. Диалоговое формирование базы клиентов

База формируется в ходе обычной работы.

При упоминании клиента агент:

1. нормализует введённое обозначение;
2. ищет точные совпадения;
3. ищет совпадения по внутренним алиасам;
4. ищет похожие варианты и возможные опечатки;
5. показывает кандидатов без раскрытия лишних данных;
6. просит подтвердить существующую карточку либо создание новой;
7. уточняет публичное имя;
8. отдельно уточняет внутреннее обозначение мастера;
9. запрашивает минимальный контакт, позволяя временно пропустить его;
10. создаёт карточку и продолжает исходную запись.

Автоматически объединять похожие карточки запрещено.

## 4. Публичное имя и внутреннее обозначение

Каждая карточка разделяет:

- `public_name` — допустимое имя для будущего обращения к клиенту;
- `internal_alias` — привычное обозначение мастера для поиска;
- `internal_notes` — закрытые рабочие заметки.

Внутренние обозначения и заметки не должны использоваться в клиентских сообщениях, публичных шаблонах или интеграциях.

Пример синтетических данных:

```text
public_name: Клиент A
internal_alias: Клиент A — увеличенное время
internal_notes: Добавлять 30 минут к стандартной длительности
```

## 5. Перенос записи

Агент находит запись, показывает её параметры, предлагает или проверяет новое время, затем после подтверждения обновляет базу и событие календаря атомарно либо выполняет компенсирующее восстановление при ошибке интеграции.

## 6. Отмена записи

Агент показывает найденную запись и запрашивает подтверждение. Запись получает статус отмены, а событие календаря удаляется или помечается согласно принятой политике. История действия остаётся в аудите.

## 7. Блокировка времени

Мастер может закрыть интервал, день или диапазон дат. Блокировки участвуют в расчёте доступности наравне с записями.

## 8. Просмотр расписания

Поддерживаются запросы:

- на день;
- на неделю;
- на выбранный период;
- по клиенту;
- по услуге;
- список свободных окон.

## 9. Административные процессы

Администратор управляет:

- доступом пользователей;
- услугами, стоимостью и длительностью;
- рабочим графиком и буферами;
- техническими настройками;
- аудитом и диагностикой;
- объединением подтверждённых дублей карточек.
