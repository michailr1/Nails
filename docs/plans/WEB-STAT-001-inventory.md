# WEB-STAT-001 — проверенная инвентаризация перед реализацией

Дата: 21 июля 2026 года.

Связанный umbrella issue: #174.

## Вывод

Отдельный Slice A с новой моделью результата визита **не нужен**: требуемая доменная основа уже реализована и принята в production в ADR-007 Slice D.

Существующие `Booking.status`, `completed_at`, `price_amount`, `price_source`, `price_confirmed_at` и immutable snapshots являются источником истины для статистики. Новую таблицу завершённых визитов, второй набор статусов или отдельную платёжную сущность не создавать.

## Существующая модель результата визита

`BookingStatus` уже содержит:

- `scheduled`;
- `completed`;
- `cancelled`;
- `no_show`.

Доменный `finalize_booking` уже обеспечивает:

- owner-scoped финализацию;
- исход `completed` или `no_show`;
- запрет финализации будущей и отменённой записи;
- идемпотентный повтор;
- позднюю подтверждённую корректировку;
- audit;
- безопасную семантику fixed/range/per-unit/on-request без ложного нуля.

Вечерний digest уже выбирает завершившиеся по времени `scheduled` записи и предлагает мастеру финализацию. Новый timer или cron для базовой статистики не нужен.

## Источник суммы

Для фактической выручки учитывать только `Booking.status == completed`.

Сумма записи берётся из финального состояния самой записи:

- подтверждённая fixed/manual сумма учитывается;
- `final_range_lower_bound_unconfirmed` не должна молча считаться подтверждённой выручкой: вернуть её отдельно как неполные данные либо ориентир;
- `final_price_unknown` не считать нулём;
- `no_show`, `cancelled` и `scheduled` не включать в фактическую выручку.

История не зависит от текущего прайса: использовать `price_amount`, `price_source`, `price_confirmed_at`, `catalog_items_snapshot` и остальные snapshot-поля записи.

## Состав записи

`catalog_items_snapshot` хранит состав основной процедуры и дополнений на момент записи. Агрегация статистики должна использовать snapshot-названия и snapshot-цены, а не текущие строки `services`.

Общая выручка считается один раз по итоговой цене Booking. Разрез по процедурам и дополнениям является аналитическим распределением и не должен повторно прибавляться к общей сумме.

Для первой версии:

- количество процедур и дополнений можно считать точно по snapshot;
- выручку по основной процедуре можно относить к итоговой сумме записи;
- выручку конкретного дополнения показывать только когда snapshot содержит достаточную однозначную сумму;
- при ручном изменении итоговой цены не выдумывать распределение разницы между компонентами.

## Существующие web-механизмы

Кабинет уже имеет:

- same-origin BFF `/web/api`;
- owner-scoped web session identity;
- календарь с диапазоном дат;
- карточки клиенток;
- «Мой прайс»;
- CSV/XLSX выгрузку календаря за период и целиком;
- CSV/XLSX выгрузку клиенток.

Ранее зафиксированное ограничение «не писать про выгрузку клиенток» более неактуально: endpoint `/web/api/exports/clients` существует. LANDING-001 должен учитывать подтверждённый экспорт календаря и клиенток.

## Timezone

Периоды статистики определяются в timezone мастера через существующий `app_timezone()`; текущая production timezone — `Europe/Moscow`.

Границы локальных дат переводятся в UTC только для запроса к `starts_at`. Нельзя группировать календарные дни напрямую по UTC-дате.

## Первый реализационный slice

Первым PR сделать read-only вертикальный slice без миграции:

1. owner-scoped Statistics API;
2. день / неделя / месяц / произвольный диапазон;
3. фактическая подтверждённая выручка;
4. состоявшиеся визиты, средний чек, уникальные клиентки;
5. cancelled/no-show и неполные цены отдельными счётчиками;
6. временной ряд по локальным дням;
7. процедуры и дополнения по количеству;
8. клиентки по выручке и визитам;
9. backend unit/integration tests на snapshots, timezone и isolation.

Web-дашборды выполнять следующим PR после стабилизации JSON-контракта.

## Не создавать

- новую миграцию только ради статистики;
- вторую модель завершения визита;
- materialized aggregates или cache на текущем масштабе;
- новый scheduler;
- отдельный analytics service/container;
- налоговый или бухгалтерский учёт;
- распределение ручной итоговой скидки/наценки по компонентам без фактических данных.
