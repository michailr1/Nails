# Ревью проекта — 2026-07-12

- Статус: принято
- Решения перенесены в `roadmap.md`, `architecture.md`, `business-processes.md`, `privacy-and-data-model.md`

## 1. Общая оценка идеи

Идея жизнеспособная. Сильные стороны:

- автоматизируется внутренняя рутина мастера, а не публичная онлайн-запись, где высокая конкуренция готовых сервисов;
- ADR-001 (перенос публичного контура на следующий этап) снижает поверхность атаки и ускоряет пилот;
- архитектура безопасности продумана заранее: Booking API как единственная точка бизнес-правил, роль из transport context, идемпотентность, аудит, подтверждения опасных операций;
- разделение `public_name` / `internal_alias` / `internal_notes` заложено в модель данных с первого дня;
- формирование клиентской базы в диалоге снимает барьер ручного переноса записной книжки.

## 2. Выявленные риски

1. **Поздняя первая польза.** Линейный roadmap по слоям откладывал живой диалог с ботом до поздних этапов, а для проекта с одним пользователем ранняя обратная связь критична.
2. **Google Calendar как самая хрупкая часть MVP.** Двусторонняя атомарная синхронизация сложна, при этом PostgreSQL уже является источником истины.
3. **Просмотр дня и недели текстом** — отдельная UX-задача, не зафиксированная как фича.
4. **Отсутствуют бытовые сценарии мастера:** повторная запись «как обычно», неявка клиентки (no-show), фактическое удлинение процедуры.
5. **Резервное копирование не упомянуто.** Для системы, заменяющей записную книжку, потеря базы означает потерю бизнеса.
6. **Часовой пояс.** Хранение и расчёты в UTC для локального бизнеса опаснее локальной зоны мастера; ошибка на час мгновенно подрывает доверие.

## 3. Принятые решения по функциям

Добавляются в план:

- «записать как обычно» — повтор последней услуги клиентки с персональной длительностью;
- персональные корректировки длительности как структурированное поле (`client_service_overrides`), а не текст внутренней заметки;
- статус no-show у записи и счётчик по клиентке;
- утренняя сводка расписания дня по расписанию;
- компактный текстовый вид дня и недели как отдельная проработанная фича;
- итоги дня и недели по выручке;
- кнопка «не то» (👎) под ответами бота с записью контекста диалога в защищённый лог разбора.

Упрощаются:

- Google Calendar — односторонний экспорт из БД best-effort с фоновой сверкой вместо атомарной двусторонней синхронизации; переносится ближе к концу MVP;
- объединение дублей карточек — вне MVP, до пилота достаточно ручной административной операции.

## 4. Принятые решения по плану

Roadmap перестроен с горизонтальных слоёв на вертикальные срезы:

1. сквозной happy path (минимальная БД, поиск окна, создание записи, просмотр дня, бот с allowlist) — как можно раньше даёт мастеру живой диалог;
2. управление записями (переносы, отмены, блокировки, подтверждения, идемпотентность, кнопка 👎);
3. клиентская база (карточки, алиасы, заметки, похожие имена);
4. повседневные сценарии (сводка, «как обычно», no-show, итоги);
5. Google Calendar (односторонний экспорт), затем пилот-сверка.

В требования первого среза добавлены:

- бэкапы PostgreSQL с регулярной проверкой восстановления;
- фиксация локального часового пояса мастера в настройках и правило вести всё расписание в нём.

В тестирование добавлены сценарии: границы рабочего дня, блокировка через полночь, одинаковые публичные имена у разных карточек, недоступность календаря не блокирует работу с записями.

## 5. Рассмотрено и отклонено

- Wizard of Oz (ручная имитация бота до кода) — требует постоянного ручного участия, не подходит;
- shadow mode и поэтапное включение write-операций через feature flags — пилот остаётся в формате параллельного ведения (NAILS-008);
- еженедельный ритуал разбора с мастером и формальные метрики пилота — обратная связь собирается через кнопку 👎 и лог нераспознанных запросов.
