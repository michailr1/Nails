# Инженерные принципы — вводная для LLM-архитектора и разработчика

Обязательно к прочтению вместе с `AGENTS.md` перед проектированием или изменением кода, ops и процессов. Эти правила существуют, потому что LLM-разработке свойственен конкретный failure mode: **при ошибке добавлять новый защитный слой вместо удаления причины**. Так на проект из двух контейнеров накопилось ~3500 строк деплой-bash, и слои начали ломаться друг о друга (см. ADR-003).

## 1. Масштаб проекта — константа решений

Nails — это **один мастер, один Telegram-бот, один VPS, два контейнера**. Любое решение проверяй вопросом: «соразмерно ли это одному мастеру?» Multi-tenant, очереди, оркестраторы, микросервисы, отдельные окружения — не для этого этапа. Если решение оправдано только «на будущее» — оно не оправдано.

## 2. Что священно и не упрощается

Ядро безопасности — единственная часть, где избыточность оправдана:

- узкие restricted tools вместо универсального доступа;
- identity только из доверенного transport context;
- подтверждение мастером каждой мутации;
- снимки цен в записях; приватность внутренних пометок и audit;
- бэкап базы перед мутирующим деплоем; backup/restore-test до пилота;
- честность ассистента: успех только после `ok=true`.

Эти инварианты не обсуждаются и не «оптимизируются».

## 3. Правила против разрастания

1. **Устраняй класс ошибки, а не симптом.** Прежде чем добавить проверку, retry или защитный слой — докажи (в описании PR), что причину нельзя удалить. Пример-эталон: вместо ручного списка маршрутов — SHA, зашитый в образ.
2. **Правило вычитания.** Добавляешь ops-файл, workflow или документ — укажи в PR, что удаляешь взамен. Суммарный объём ops и process-кода не должен расти от релиза к релизу.
3. **Никаких клонов V2/V3.** Скрипт чинится на месте; версии — это git history. Появление файла `*-v2.*` — сигнал, что нарушен пункт 1.
4. **Один постоянный deploy-скрипт** (`ops/deploy/deploy.sh`, ADR-003). Одноразовые релизные runbook'и запрещены.
5. **Production validation предшествует merge.** VPS проверяет точный PR-head SHA, не меняя checkout. Основной агент затем fast-forward’ом мержит тот же SHA. Squash/rebase/merge commit для production release запрещены.
6. **Верифицируй по источнику истины.** Checkout SHA, candidate source ref, built `NAILS_GIT_SHA`, running `NAILS_GIT_SHA` и merged SHA должны быть названы явно. Жёстко заданные baseline из прошлых запусков запрещены.
7. **Rollback = deploy предыдущего SHA.** Отдельная rollback-машинерия не строится. Следствие: миграции обратно-совместимы минимум на один релиз.
8. **Бюджет изменения ops — 100 строк.** Если diff по `ops/` больше, остановись и вынеси в PR-описание вопрос «что здесь можно не делать?».
9. **Диагностика прод-инцидента начинается с тождества**: тот ли commit/ref, из того ли дерева собран image, тот ли SHA в running container. Только потом — всё остальное.
10. **Один источник для факта.** Прежде чем создать документ, проверь, не описан ли факт в `status.md`, `context/current.md`, roadmap или operations-контрактах. Дублирование фактов в нескольких документах — источник расхождений.

## 4. Дисциплина изменений (для каждой задачи)

По мотивам практики coding-агентов (наблюдения А. Карпати в сторонней адаптации). Применяется к основному агенту и любому coding/review-агенту; полномочий VPS-агента не расширяет.

**До изменения кода** зафиксируй в описании PR (шаблон PR требует эти поля): цель; что сознательно не входит в задачу; существенные допущения; критерии приёмки и способ проверки каждого.

**Проверка фактов до проектирования.** Перед любым архитектурным предложением или новым механизмом перечисли существующие функции, эндпоинты и правила, которые уже решают часть задачи, — со ссылками на файлы (шаблон PR требует поле «Существующие механизмы»). Пустой список означает «поиск не выполнялся» и не принимается. Предложение механизма, дублирующего существующий, отклоняется на ревью. Причина правила: LLM-агенту спроектировать новое дешевле, чем найти своё же старое, — известный случай: предложение добавить `exclude_booking_id`, уже реализованный автором предложения.

**Ревью границ абстракций.** Архитектурное предложение — новый слой, stateful-компонент, хранилище состояния, планировщик, обобщающий механизм — до реализации проходит независимое ревью соразмерности. Ревьюер сверяет предложение с фактическим кодом и вправе заменить механизм на более дешёвый способ (правило skill/контракта, существующая идемпотентность, поле в существующей таблице).

**Неоднозначность.** Если выбор влияет на данные, схему, публичный API, безопасность или поведение пользователя — остановись и разреши её явно до реализации. В остальных случаях зафиксируй разумное допущение в PR и продолжай. Молчаливый выбор реализации запрещён.

**Хирургические изменения.** Каждая изменённая строка связана с целью задачи. Запрещены: попутный рефакторинг, переформатирование соседнего кода, переписывание понятных комментариев, удаление старого неиспользуемого кода вне задачи. Удалять можно только то, что стало ненужным непосредственно из-за твоего изменения. Обнаруженная сторонняя проблема — отдельное issue, а не расширение текущего diff.

**Дефекты.** Сначала проверка, воспроизводящая дефект, затем исправление, затем regression-тест в том же PR. Если воспроизводящая проверка невозможна — объясни в PR почему.

**CI-падения.** Чинятся только после локального воспроизведения той же командой, что в CI; правка вносится в файл и строку, которые назвал инструмент, а не в предполагаемый. Ошибки форматирования и порядка импортов исправляются автофиксером (`ruff check --fix`), а не вручную и не подавлением через `noqa`: ручная сортировка импортов сама порождает новые ошибки того же класса.

**Перед завершением**: прогони целевые проверки изменённого и применимые regression-тесты; просмотри итоговый diff на посторонние изменения; сверь результат с каждым критерием приёмки; явно назови то, что осталось непроверенным.

## 5. Куда двигаться (порядок работ)

1. Завершить переход на pre-merge candidate validation и постоянный `deploy.sh`.
2. NAILS-002F: автоматические бэкапы + подтверждённый restore-test — **жёсткий барьер** перед реальными данными.
3. Пилот с мастером; обратная связь через 👎 и лог нераспознанных запросов.
4. Только после стабильного пилота — NAILS-003+ (переносы/отмены), затем остальное по roadmap.

Новые большие подсистемы (Google Calendar, публичный клиентский бот, multi-master) не начинаются, пока мастер не живёт на системе. Клиентский контур, когда дойдёт очередь, — детерминированный бот без LLM (ADR-004): узкая недоверенная аудитория не получает разговорную LLM-поверхность.

## 6. Чек-лист перед PR

- [ ] Решение соразмерно «одному мастеру»?
- [ ] Существующие механизмы перечислены, дублей не создаётся?
- [ ] Удалена причина, а не прикрыт симптом?
- [ ] Что удалено взамен добавленного?
- [ ] Не появился ли клон существующего скрипта/документа?
- [ ] Инварианты из раздела 2 не ослаблены?
- [ ] Каждая изменённая строка связана с целью задачи?
- [ ] Candidate SHA и будущий merge SHA идентичны?
- [ ] Критерии приёмки из описания PR проверены, непроверенное названо явно?
